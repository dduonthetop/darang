# 리팩토링 메모

## 목표
- 전체 구조를 더 읽기 쉽게 정리한다.
- 배포 경로와 실제 작업 경로가 분리된 현재 구조에서 중복을 줄인다.
- 리팩토링 중 기능 변화가 생기지 않도록 사이드 이펙트를 계속 점검한다.

## 페르소나
- PM: 배포 구조가 흔들리지 않는지, 운영자가 계속 같은 방식으로 쓸 수 있는지 본다.
- 프론트 엔지니어: 화면 동작은 유지하면서 상태 관리와 렌더링 흐름을 단순화한다.
- 백엔드 엔지니어: 경로, 저장, 동기화 로직의 중복을 줄이고 변경 지점을 명확하게 만든다.
- UI/UX 디자이너: 관리자 화면의 문맥이 유지되고 상호작용 피드백이 깨지지 않는지 본다.
- 사용자: 로그인, FAQ 조회/수정, 공개 FAQ 사용 흐름이 이전과 같아야 한다.

## 1차 토론
- PM: 이번 턴에서는 전면 재설계보다 안전한 구조 정리가 우선이다.
- 프론트 엔지니어: `admin.js`는 상태 변경과 DOM 갱신이 섞여 있어 작은 수정에도 부담이 크다.
- 백엔드 엔지니어: 배포 대상 경로가 여러 파일에 흩어져 있어 누락 위험이 있다.
- UI/UX 디자이너: 문구와 상호작용 흐름은 유지하고 내부 구조만 정리하는 편이 안전하다.
- 사용자: 저장 후 목록, 세션, 공개 FAQ가 그대로 동작하면 좋다.

## 현재 관찰
- 실제 앱 소스는 `darang/faq` 아래에 모여 있다.
- 루트의 `app.py`는 래퍼이고, 루트 HTML/정적 파일은 배포본 역할을 한다.
- 관리자 저장 시 정적 FAQ 데이터와 GitHub 동기화가 함께 엮여 있다.
- 배포 복사 경로 정의가 여러 곳에서 반복된다.

## 사이드 이펙트 체크리스트
- `/chat` 응답 형식이 바뀌지 않는가
- `/admin/api/faqs` 조회/저장 형식이 바뀌지 않는가
- `local_faq_data.js` 생성 위치가 유지되는가
- 루트 배포본과 `darang/faq` 작업본의 관계가 깨지지 않는가
- 관리자 화면에서 검색, 새 항목, 삭제, 저장, 로그아웃 흐름이 유지되는가

# iPark FAQ Chatbot BRD

## 1. 프로젝트 목적
브랜드 운영 정보를 통합 FAQ 챗봇으로 구축하여 반복 문의 응답 시간을 단축하고, 바이어/운영팀의 안내 문구를 표준화한다.

## 2. 문제 정의
- 브랜드 입점/운영 문의가 카카오톡, 전화, 메일로 분산되어 응답 품질이 일관되지 않다.
- 동일 질문 반복으로 MD/운영 담당자의 업무부하가 높다.
- 정책 변경 시 최신 기준 반영이 지연되어 오안내 위험이 발생한다.

## 3. 대상 사용자
- 브랜드: 브랜드/협력사 관점에서 묻는 질문
- 바이어: 내부 바이어가 응대 또는 운영 기준 확인을 위해 보는 질문
- 공통: 양측 모두에게 동일하게 적용 가능한 질문

## 4. 챗봇 기능 정의
- FAQ 기반 질의응답 (입점문의/입점진행/매장운영)
- 질문 의도 분류 후 단계별 답변 제공
- RAG 검색(CSV + 카카오톡 정제 데이터) 기반 답변 생성
- 불확실/정책충돌 시 `담당 MD 확인 필요` 예외 안내
- 운영 로그 기반 FAQ 데이터셋 지속 업데이트

## 5. 질문 단계 구조
### 입점문의
- 입점 가능 여부, 제안서 제출, 팝업 가능 여부, 기본 운영시간

### 입점진행
- 계약/서류, 출입신청, 차량등록, 하역, 설치/철수 준비, 광고물 제출

### 매장운영
- POS 사용/장애, 쇼핑백/비품, 정산/세금계산서, 운영 이슈 대응

## 6. FAQ 데이터 구조
- 기본 단위: 1 질문(의도) : 1 표준 답변
- FAQ는 단순 문장 모음이 아니라, 향후 GPT/RAG/CSV 데이터셋으로 변환 가능한 구조로 관리한다.
- 지식베이스 항목은 질문-답변 외에도 대상 사용자, 후속 액션, 근거 출처, 신뢰도 유형을 포함해 운영한다.

## 6.1 챗봇 지식베이스 데이터 스키마
아래 스키마는 FAQ 원문을 GPTs/RAG/CSV로 변환할 때 사용하는 기준 필드다.

- `faq_id`: FAQ 항목의 고유 식별자. 문서/CSV/테스트 간 동일 항목 추적용.
- `stage`: 질문 단계. `입점문의 / 입점진행 / 매장운영` 중 1개로 분류.
- `audience`: 질문 관점 구분. `브랜드 / 바이어 / 공통`으로 응답 대상 명확화.
- `category`: 업무 카테고리(예: 출입, 주차, POS, 정산). 검색 및 필터링 기준.
- `question`: 대표 질문 원문.
- `paraphrases`(중요): 같은 의미의 질문을 다양한 표현으로 저장해 검색 정확도를 높이기 위한 필드.
- `answer`: 표준 답변 원문.
- `next_action`(중요): 답변 후 사용자가 취해야 할 다음 행동(신청, 제출, 문의) 안내를 위한 필드.
- `contact_channel`: 담당 부서/연락처/시스템 경로 등 문의 채널.
- `restrictions`: 시간/장소/조건/예외 등 제한사항.
- `visibility`: 공개 범위(외부공개/내부전용 등).
- `confidence_type`(중요): 답변 신뢰 유형.
  - 확정 정책형 답변
  - 운영 사례형 답변
  - 확인 필요형 답변
  오답과 과잉 단정을 줄이기 위해 응답 생성 시 필수 참조한다.
- `updated_at`: 최종 갱신일(YYYY-MM-DD).
- `source`: 근거 문서/로그(FAQ 문서, 카카오톡 로그, 매뉴얼 등).

### 질문 표현 다양성 규칙
각 FAQ 항목은 실제 사용자 질문 다양성을 반영하기 위해, 대표 질문(`question`) 외에 최소 3개 이상의 `paraphrases`(동의 질문 표현)를 포함한다.

`paraphrases` 작성 기준:
- 동일 의미의 질문을 자연어 표현으로 작성
- 키워드 축약형 질문 포함
- 사용자 실제 문의 표현 반영

예시:
- `question`: POS는 기본 제공되나요?
- `paraphrases`:
  - 포스 제공되나요
  - 계산용 POS 있나요
  - POS 장비 지원되나요
  - 포스 기본 제공되는지

## 7. 응답 정책
- 단순 키워드 매칭보다 업무 맥락을 우선 해석한다.
- 관련 근거가 2개 이상이면 결합해 구조화 답변한다.
- 답변 형식은 `단계 → 핵심안내 → 필요조건/자료 → 다음 액션` 순으로 작성한다.
- `audience`가 브랜드면 안내형, 바이어면 업무형, 공통이면 중립형 톤을 사용한다.
- 응답 생성 시 `confidence_type`을 반드시 참조해 단정 수준을 조절한다.
- `confidence_type`은 자동분류 후 바이어 수동검수로 확정한다(자동분류 단독 배포 금지).

## 8. 예외 처리 정책
- 아래 항목은 단정하지 않고 확인 필요로 안내한다.
  - 확정 수수료율/계약 세부조건
  - 실시간 운영시간 변동/임시 정책
  - 현장별 예외 동선/차량 제한
  - 하드웨어 호환성, 연동 기술사양
- 충돌 데이터 발생 시 최신 확정 데이터셋(CSV/입력표) 우선 적용

## 8.1 리스크 및 오류 관리 답변 정책
- 확정 정책형 답변(Policy): 문서/시스템 기준으로 고정 안내 가능한 답변
- 운영 사례형 답변(Guideline): 운영 기준이 있는 실제 사례 기반이나 현장별 변동 가능성이 있는 답변
- 확인 필요형 답변(Approval Required): 담당 바이어/시설/주차/보안 최종 확인이 필요한 답변

적용 원칙:
- 챗봇은 답변 생성 시 위 3개 유형 중 하나로 내부 분류한 뒤 응답한다.
- `Policy`는 단정형으로 안내하되, 기준 문서/시스템 경로를 함께 제시한다.
- `Guideline`은 사례 기반임을 명시하고 변동 가능성을 문장에 포함한다.
- `Approval Required`는 단정 답변을 피하고 확인 주체(바이어/시설/주차/보안)를 명시한다.
- 정책 충돌 또는 근거 부족 시 `Approval Required`로 상향해 오안내를 방지한다.

## 9. 시스템 연계
- 질문 입력: GPTs 대화 UI(링크형)
- 단계 분류: Instructions 룰 기반 분류
- RAG 검색: CSV + 카카오톡 정제 로그
- 답변 생성: LLM
- 운영 데이터 관리: `references/source_answers.xls` → CSV 변환 → GPTs/RAG 반영

## 9.1 공식 시스템 맵
- 전자계약: 프리미엄 나이스다큐
- 대금정산: 트러스빌
- 출입신청: 파트너스 웹사이트(PMS) `https://pms.iparkmall.co.kr/intro`
- 매출확인: 파트너스 웹사이트(PMS) `https://pms.iparkmall.co.kr/intro`

### 운영 연락처
- 내선전화 신청: KT `02-2012-2200`
- 보안: `02-2012-4112`
- 방재: `02-2012-0119`
- 전기: `02-2012-0087`
- 시설: `02-2012-0070`
- 건축: `02-2012-0056`
- 미화: `02-2012-0058`
- 주차: `02-2012-0074`
- 전산/POS: `02-2012-0200`
- 교환실: `02-2012-0101`
- 고객 상담실: `02-2012-0000`, `02-2012-0002`

### 아이파크몰 건물명
- 리빙파크
- 패션파크
- 테이스트파크
- 더센터

## 9.2 contact_channel 매핑 규칙
- FAQ 데이터셋의 `contact_channel`은 질문 카테고리 기준으로 표준값을 매핑한다.
- 하나의 질문이 다부서 이슈를 포함할 경우 `1차 문의 채널`을 앞에, `에스컬레이션 채널`을 뒤에 기재한다.
- 연락처 변경 시 BRD 시스템 맵과 dataset의 `contact_channel`을 동시에 갱신한다.

## 10. 테스트 기준
- 기능 테스트: 3단계 분류 정확도, 검색 결과 매칭, 답변 생성
- 품질 테스트:
  - 핵심 질문군(운영시간/출입/주차/POS/정산) 정확 응답
  - 복합질문(위치+시간, 채널+서류) 누락 없이 응답
  - 예외 상황에서 `확인 필요` 문구 적용
- 수용 기준:
  - 핵심 질문셋 50개 이상 답변 가능
  - 오답/모호답 비율 내부 기준 이하

## 11. 향후 확장
